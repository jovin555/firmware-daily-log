## What I Explored Today

Today I dug into libbpf and Compile-Once Run-Everywhere (CO-RE) — the modern approach to writing portable eBPF programs in C. For months I'd been fighting with kernel version-specific BTF headers and manual relocation tables. libbpf + CO-RE eliminates that pain entirely. I built a minimal kprobe that reads `struct task_struct` fields without hardcoding offsets, and it ran identically on a 5.15 kernel and a 6.1 kernel without recompilation. This is the future of eBPF development.

## The Core Concept

The fundamental problem with classic eBPF (BCC-style) is that every kernel version shifts internal struct layouts. A `struct task_struct->pid` offset on kernel 5.10 is different on 5.15. BCC handles this by compiling BPF programs at runtime on the target machine, pulling in the local kernel headers. That works, but it requires LLVM/clang on the target, adds startup latency, and bloats deployment.

CO-RE solves this by separating compilation from relocation. You compile your BPF program once against a single set of kernel headers (typically from a "base" kernel like 5.4 or 5.10). The compiled ELF contains BPF CO-RE relocations — metadata that records which field accesses need adjustment. At load time, libbpf reads the target kernel's BTF (BPF Type Format, available since 5.2 with `CONFIG_DEBUG_INFO_BTF=y`), computes the correct offsets, and patches the bytecode before the verifier sees it.

The key insight: you don't need kernel headers on the target. You only need BTF, which is either embedded in the kernel (vmlinux) or shipped as a separate `btfhub` package. This makes eBPF deployments as simple as shipping a single `.bpf.o` ELF file and a small loader binary linked against libbpf.

## Key Commands / Configuration / Code

### Step 1: Generate vmlinux.h (once per base kernel)

```bash
# Requires CONFIG_DEBUG_INFO_BTF=y on the build machine
bpftool btf dump file /sys/kernel/btf/vmlinux format c > vmlinux.h
```

This produces a single header with all kernel type definitions. Your BPF program includes this instead of `<linux/sched.h>` etc.

### Step 2: Write a CO-RE aware BPF program

```c
// task_pid_kprobe.bpf.c
#include "vmlinux.h"               // Kernel types from BTF
#include <bpf/bpf_helpers.h>
#include <bpf/bpf_tracing.h>

char LICENSE[] SEC("license") = "GPL";

SEC("kprobe/sys_clone")
int kprobe__sys_clone(struct pt_regs *ctx)
{
    // CO-RE: libbpf will relocate the field offsets at load time
    struct task_struct *task = (struct task_struct *)bpf_get_current_task();
    
    // Safe field access — no hardcoded offsets
    u32 pid = BPF_CORE_READ(task, tgid);
    u32 ppid = BPF_CORE_READ(task, real_parent, tgid);
    
    bpf_printk("CO-RE: pid=%d, ppid=%d\n", pid, ppid);
    return 0;
}
```

Note `BPF_CORE_READ` — this macro generates the relocation record. Never use direct pointer dereferences like `task->tgid`; they won't relocate.

### Step 3: Compile with CO-RE support

```bash
clang -g -O2 -target bpf -D__TARGET_ARCH_x86 \
    -I/usr/include/x86_64-linux-gnu \
    -c task_pid_kprobe.bpf.c -o task_pid_kprobe.bpf.o

# Verify CO-RE relocations exist
bpftool gen skeleton task_pid_kprobe.bpf.o
# Look for ".BTF" and ".BTF.ext" sections
```

### Step 4: Load with libbpf (C loader)

```c
// loader.c
#include <bpf/libbpf.h>
#include "task_pid_kprobe.skel.h"  // Generated by bpftool gen skeleton

int main() {
    struct task_pid_kprobe_bpf *skel;
    int err;

    // libbpf handles BTF lookup and relocation automatically
    skel = task_pid_kprobe_bpf__open_and_load();
    if (!skel) { /* error */ }

    err = task_pid_kprobe_bpf__attach(skel);
    if (err) { /* error */ }

    printf("eBPF program loaded with CO-RE relocations\n");
    // Keep running...
    task_pid_kprobe_bpf__destroy(skel);
    return 0;
}
```

Compile the loader:
```bash
gcc -o loader loader.c -lbpf -lelf -lz
```

## Common Pitfalls & Gotchas

**1. Missing BTF on target kernel**
The most common deployment failure. Check with `cat /sys/kernel/btf/vmlinux | wc -c`. If it's 0 or the file doesn't exist, you need `CONFIG_DEBUG_INFO_BTF=y`. For older kernels (pre-5.2), use BTFHub to ship BTF files alongside your program. libbpf supports `bpf_object__load_btf()` with custom BTF.

**2. Using direct struct access instead of BPF_CORE_READ**
I wasted two hours debugging a segfault because I wrote `task->tgid` instead of `BPF_CORE_READ(task, tgid)`. Direct access compiles fine but produces no relocation record. On a different kernel, the offset is wrong and you read garbage (or crash the verifier). Always use `BPF_CORE_READ` for any field access, including nested structs: `BPF_CORE_READ(task, real_parent, tgid)`.

**3. Forgetting `-g` flag during compilation**
The `-g` flag emits DWARF debug info that bpftool uses to generate the BTF relocation sections. Without it, your `.o` file has no `.BTF` section and libbpf can't relocate. The program loads but silently reads wrong offsets. Always compile BPF programs with `-g -O2`.

## Try It Yourself

1. **Build a CO-RE kprobe that reads `current->cred->uid`** — use `BPF_CORE_READ` with nested structs. Verify it prints the correct UID on two different kernel versions without recompilation.

2. **Deploy without BTF on target** — compile on a machine with BTF, then copy the `.bpf.o` and loader to a kernel without BTF. Use `btfhub` to provide the BTF file. Modify the loader to call `bpf_object__load_btf()` with the custom BTF.

3. **Add a CO-RE relocation check** — write a small loader that prints all relocation entries using `bpf_object__for_each_program()` and `bpf_program__relocatable()`. Verify that every field access has a corresponding relocation record.
