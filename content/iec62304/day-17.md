## What I Explored Today

Today I dug into the practical mechanics of regression testing under IEC 62304, specifically how to tie it to a change control process that doesn't collapse under its own weight. The standard doesn't mandate a specific tool or framework, but clause 5.8.3 (Change Control Process) and 6.4 (Regression Test Plan) together demand that every change to a software unit, integration, or system be accompanied by a demonstrable set of re-run tests. I focused on how to implement this with a real CI pipeline, a traceability matrix, and a minimal but effective test selection strategy.

## The Core Concept

The "why" behind regression testing in IEC 62304 is not about catching bugs—it's about proving that the *unchanged* parts of the system still satisfy their safety requirements after a change. The standard treats the software as a set of interacting safety-critical functions. When you modify a module, you risk breaking the interfaces, timing, or data flows that other modules depend on. The change control process (clause 5.8.3) requires you to document the change, assess its impact, and then execute a regression test plan that covers all affected areas. The trap is running the full test suite on every commit—that's expensive and often unnecessary. Instead, the standard expects you to *justify* your regression scope. A good approach is to use a requirements-to-test traceability matrix (RTM) and a dependency graph to select only the tests that exercise changed code paths or their consumers.

## Key Commands / Configuration / Code

Below is a practical example of a CI pipeline step that runs targeted regression tests based on a change impact analysis file. This assumes you use pytest for unit/integration tests and a JSON file that maps source files to test files.

**File: `impact_map.json`** (generated by a static analysis tool or manually maintained)
```json
{
  "src/pump_controller.c": ["tests/test_pump_controller.py", "tests/test_safety_monitor.py"],
  "src/dose_calculator.c": ["tests/test_dose_calculator.py", "tests/test_alarm_manager.py"],
  "src/safety_monitor.c": ["tests/test_safety_monitor.py"]
}
```

**File: `ci/run_regression.sh`**
```bash
#!/bin/bash
# Run regression tests based on changed files in the current PR/commit
# Usage: ./ci/run_regression.sh <list_of_changed_files>

CHANGED_FILES="$@"
TEST_FILES=""

# Collect all test files that map to changed source files
for file in $CHANGED_FILES; do
    # Look up the file in impact_map.json using jq
    # jq -r outputs raw strings, @csv joins them with newlines
    mapped_tests=$(jq -r --arg f "$file" '.[$f] // empty | @csv' impact_map.json)
    if [ -n "$mapped_tests" ]; then
        # Remove quotes from CSV and append to list
        mapped_tests_clean=$(echo "$mapped_tests" | tr -d '"')
        TEST_FILES="$TEST_FILES $mapped_tests_clean"
    fi
done

# Deduplicate and run
UNIQUE_TESTS=$(echo "$TEST_FILES" | tr ' ' '\n' | sort -u | tr '\n' ' ')
if [ -n "$UNIQUE_TESTS" ]; then
    echo "Running regression tests: $UNIQUE_TESTS"
    pytest -v --junitxml=regression_results.xml $UNIQUE_TESTS
else
    echo "No regression tests mapped for changed files. Running full suite as fallback."
    pytest -v --junitxml=regression_results.xml tests/
fi
```

**Example CI config snippet (GitHub Actions):**
```yaml
- name: Run targeted regression tests
  run: |
    # Get list of changed files from the PR
    CHANGED=$(git diff --name-only origin/main...HEAD)
    ./ci/run_regression.sh $CHANGED
```

**Traceability matrix snippet (CSV format for audit):**
```csv
Requirement_ID,Source_File,Test_File,Test_Status
REQ-PUMP-001,src/pump_controller.c,tests/test_pump_controller.py,PASS
REQ-PUMP-002,src/pump_controller.c,tests/test_pump_controller.py,PASS
REQ-SAFE-001,src/safety_monitor.c,tests/test_safety_monitor.py,PASS
REQ-DOSE-001,src/dose_calculator.c,tests/test_dose_calculator.py,PASS
```

The key is that the `impact_map.json` must be updated whenever a new source file or test is added. This is part of the change control process itself—every change to the test infrastructure must be reviewed and approved.

## Common Pitfalls & Gotchas

1. **Running the full test suite on every change and calling it "regression."**  
   IEC 62304 auditors will ask: "How did you determine that all tests were necessary?" If you can't justify the scope, you haven't done impact analysis. A full suite is acceptable only if you can prove the entire system is affected—which is rare. Instead, use a dependency graph or a formal impact analysis document.

2. **Forgetting to update the impact map when adding new modules.**  
   This is the most common failure I've seen. The change control process must include a step to update the traceability matrix and the test mapping. If a new source file `src/alarm_handler.c` is added but never mapped to any test, a regression run will silently skip it. The fix is to enforce a pre-merge check that validates every source file has at least one test mapping.

3. **Treating regression tests as a separate suite from unit/integration tests.**  
   This leads to duplication and maintenance burden. Instead, tag your tests with markers like `@pytest.mark.regression` or `@pytest.mark.safety` and use the same test codebase. The regression test plan is just a *selection* of existing tests, not a new set of tests.

## Try It Yourself

1. **Build your own impact map.**  
   Take a small embedded project (e.g., a thermostat controller) and create a JSON file that maps each `.c` file to its corresponding test file. Then write a script that, given a list of changed files, prints the test files that should run.

2. **Add a pre-merge CI check.**  
   In your CI pipeline, add a step that fails if any source file in the commit has no entry in the impact map. Use `jq` to check for missing keys and output a clear error message.

3. **Implement a regression test selector with pytest markers.**  
   Add `@pytest.mark.regression` to a subset of your tests. Then modify your CI to run only those markers when the change is minor (e.g., a comment fix). For larger changes, run the full suite. Document your selection criteria in a change control log.
