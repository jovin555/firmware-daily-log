## What I Explored Today

Today I dug into the layer system that makes Yocto both powerful and maddening. Layers are the fundamental unit of organization in OpenEmbedded/Yocto, and understanding how they stack, override, and interact is the difference between a clean build and a dependency hell. I spent the morning tracing how `bblayers.conf` controls layer priority, and the afternoon watching how BitBake resolves recipe conflicts when two layers provide the same recipe. The architecture is elegant, but the gotchas are real.

## The Core Concept

Think of layers as a stack of transparent overlays on an overhead projector. Each layer contains recipes, configuration, and metadata. When BitBake needs to find a recipe (say, `openssl_3.0.bb`), it searches through every enabled layer in priority order. The first match wins. This is how you can override a recipe from `meta` (the core layer) with a custom version in `meta-mycompany` without touching the original files.

But layers aren't just for recipes. They also contribute to `BBFILES`, `BBPATH`, and layer-specific configuration variables. A well-structured layer has a `conf/layer.conf` that tells BitBake what it provides, its priority, and which paths to search.

Priority is an integer. Higher number = higher priority. Default is typically 6 for core layers, but you can set it to 99 for your custom layer to ensure your recipes always win. This is critical when you have multiple layers that might define the same recipe name (e.g., `u-boot_2024.01.bb` from both `meta-ti` and `meta-myboard`).

The `bblayers.conf` file in your build directory is the master switchboard. It lists every layer BitBake should consider, and the order matters. Layers listed first are searched first for recipes, but the `BBFILE_PRIORITY` in each layer's `layer.conf` overrides list order for recipe resolution.

## Key Commands / Configuration / Code

### The bblayers.conf file (auto-generated by `oe-init-build-env`)

```bash
# Build directory: poky/build/conf/bblayers.conf
# This is the file that tells BitBake which layers to use

LCONF_VERSION = "7"

BBPATH = "${TOPDIR}"
BBFILES ?= ""

# These are the layers you've enabled
BBLAYERS ?= " \
  /home/user/poky/meta \
  /home/user/poky/meta-poky \
  /home/user/poky/meta-yocto-bsp \
  /home/user/meta-mycompany \
  "

# Order matters: meta is first, meta-mycompany is last
# But priority in layer.conf overrides this for recipe resolution
```

### A typical conf/layer.conf

```bash
# File: meta-mycompany/conf/layer.conf
# This file lives inside your custom layer

# Tell BitBake this is a layer
BBPATH .= ":${LAYERDIR}"

# Add all .bb and .bbappend files in this layer
BBFILES += "${LAYERDIR}/recipes-*/*/*.bb \
            ${LAYERDIR}/recipes-*/*/*.bbappend"

# Set the priority (higher = wins over other layers)
BBFILE_PRIORITY_meta-mycompany = "10"

# Declare dependencies on other layers (optional but good practice)
LAYERDEPENDS_meta-mycompany = "core"
LAYERRECOMMENDS_meta-mycompany = "openembedded-layer"

# Specify compatible Yocto releases
LAYERSERIES_COMPAT_meta-mycompany = "kirkstone"
```

### Checking layer priority at runtime

```bash
# See all enabled layers and their priorities
bitbake-layers show-layers

# Output example:
# layer                 path                                      priority
# ==========================================================================
# meta                  /home/user/poky/meta                      5
# meta-poky             /home/user/poky/meta-poky                 5
# meta-yocto-bsp        /home/user/poky/meta-yocto-bsp            5
# meta-mycompany        /home/user/meta-mycompany                 10

# See which layer provides a specific recipe
bitbake-layers show-recipes openssl

# Output shows the recipe path and which layer it comes from
```

### Creating a new layer from scratch

```bash
# The bitbake-layers tool can scaffold a layer for you
bitbake-layers create-layer ../meta-mycompany

# This creates:
# meta-mycompany/
# ├── conf/
# │   ├── layer.conf
# │   └── layer.conf.append (if needed)
# ├── recipes-example/
# │   └── example/
# │       └── example_0.1.bb
# └── COPYING.MIT

# Then add it to bblayers.conf
bitbake-layers add-layer ../meta-mycompany
```

## Common Pitfalls & Gotchas

### 1. Layer priority is not the same as layer order in bblayers.conf

Many engineers assume the order in `BBLAYERS` determines recipe resolution order. It does for `BBFILES` scanning, but `BBFILE_PRIORITY` in each layer's `conf/layer.conf` is what BitBake uses when two layers provide the same recipe. If you have `meta-oe` (priority 6) and `meta-mycompany` (priority 10), your layer wins even if it's listed last. Always check with `bitbake-layers show-layers` to confirm.

### 2. Missing layer dependencies cause silent failures

If your layer depends on `meta-oe` but you forget to add it to `bblayers.conf`, BitBake won't error immediately. It will fail later when it can't find a recipe your layer inherits. Use `LAYERDEPENDS` in your `layer.conf` to catch this early. The `bitbake-layers layerindex-fetch` command can help resolve dependencies.

### 3. .bbappend files can apply to recipes from layers you don't have

A `.bbappend` file in your layer targeting `recipes-kernel/linux/linux-yocto_5.15.bbappend` will silently do nothing if the base recipe isn't in any enabled layer. BitBake won't warn you. Always verify with `bitbake-layers show-appends` that your append files are actually being applied.

## Try It Yourself

1. **Inspect your current layer setup**: Run `bitbake-layers show-layers` in your build directory. Note the priorities. Then run `bitbake-layers show-recipes bash` to see which layer provides the bash recipe and what its priority is.

2. **Create a custom layer and override a recipe**: Use `bitbake-layers create-layer` to scaffold `meta-myoverrides`. Set `BBFILE_PRIORITY` to 20. Copy a simple recipe (like `hello_1.0.bb` from `meta-skeleton`) into your layer, modify the `SUMMARY` field, add the layer to `bblayers.conf`, and run `bitbake hello`. Verify your version builds.

3. **Trace a recipe conflict**: Create two layers (`meta-a` and `meta-b`) each providing `fake_1.0.bb` with different `SUMMARY` values. Set `meta-a` priority to 5 and `meta-b` to 10. Add both to `bblayers.conf`. Run `bitbake-layers show-recipes fake` and confirm `meta-b`'s version is selected. Then swap priorities and rebuild.
